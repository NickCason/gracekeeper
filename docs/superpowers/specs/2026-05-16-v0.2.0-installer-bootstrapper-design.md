# GraceKeeper v0.2.0 — Burn Bootstrapper + Dismisser Lifecycle Rework

**Status:** Approved (brainstorming session 2026-05-16)
**Author:** Nick Cason
**Target release:** v0.2.0

## Problem

Two issues in v0.1.0-rc3 motivate this release:

1. **Dismisser does not start at install time.** The rc3 attempt — a `schtasks /Create /RU [LogonUser] /IT` trampoline custom action — does not reliably produce a running `AutoHotkey64.exe` after install completes. Users must log out and log back in for the HKLM Run entry to launch it. This is the primary functional bug.
2. **Installer looks stock.** The current `WixUI_Minimal` flow with rebranded sidebar/banner bitmaps is acceptable but obviously a WiX/MSI installer. It does not match the polish of the dashboard or the perception of "real product."

A secondary, long-known limitation also blocks the dismisser from being self-healing: if `AutoHotkey64.exe` dies mid-session, nothing restarts it until the next logout/login cycle. The supervisor code in `GraceKeeper.Core` exists and is unit-tested but is not wired up.

## Goals

- Replace the bare MSI distribution with a WiX Burn bundle (`GraceKeeper-0.2.0.exe`) carrying a custom WPF Bootstrapper Application (BA) for both install and uninstall flows.
- Eliminate the install-time launch failure by having the BA process (running as the invoking user) directly start the dashboard at install completion. The dashboard then owns the dismisser.
- Wire up the dismisser supervisor inside the dashboard so a dead `AutoHotkey64.exe` is respawned within 30 seconds.
- Provide a custom uninstall UI through the same bundle, reached via Apps & Features.

## Non-Goals

- **Code signing.** SmartScreen "Windows protected your PC" remains a known limitation. Slated for v0.3.0 (pending SignPath.io Foundation approval, ~1–2 weeks lead time).
- **Per-user install path.** Per-machine to `C:\Program Files\GraceKeeper\` remains the only supported install scope.
- **Standalone MSI in release artifacts.** The MSI continues to exist inside the bundle and is buildable for development, but only the `.exe` is published. SI-shop SCCM/Intune deployment is a v0.3.0 candidate.
- **Repair / Modify ARP entries.** Disabled in the bundle. Uninstall + reinstall is the recovery story.
- **Localization / accessibility audit.** English only, default screen-reader behavior.

### Runtime — install flow

1. User double-clicks `GraceKeeper-0.2.0.exe`. Windows shows SmartScreen warning (v0.3.0). User clicks "Run anyway."
2. Burn engine starts. Loads `GraceKeeper.Bootstrapper.dll` (managed BA). BA process runs **as the invoking user**, no elevation yet.
3. BA displays Welcome page (hero, EULA, accept checkbox, Install button).
4. User accepts EULA, clicks Install. BA calls `engine.Plan(LaunchAction.Install)` then `engine.Apply()`.
5. Burn elevates and invokes `msiexec /i GraceKeeper-0.2.0.msi /qn` against the embedded MSI.
6. BA Progress page reflects live `ExecuteMsiMessage` / `Progress` events from the engine.
7. MSI installs: files to `C:\Program Files\GraceKeeper\`, ProgramData layout with ACLs, scheduled cleaner task, single HKLM Run entry for `GraceKeeper.exe`, Start Menu shortcut.
8. MSI completes. Burn returns control to BA. BA shows Finish page with "Launch GraceKeeper now" checkbox (default checked).
9. User clicks Finish. If checkbox is checked, BA calls `Process.Start("[ProgramFiles64Folder]\GraceKeeper\GraceKeeper.exe")`. BA exits.
10. `GraceKeeper.exe` starts in the user's session. Its startup logic (see Dismisser ownership below) spawns the dismisser.

### Runtime — uninstall flow

1. User opens Settings → Apps → GraceKeeper → Uninstall. Windows invokes `GraceKeeper-0.2.0.exe /uninstall` (bundle's ARP entry).
2. BA opens UninstallConfirm page: "Uninstall GraceKeeper? Your install counters will be preserved unless you check 'Remove all data'."
3. User confirms. BA calls `engine.Plan(LaunchAction.Uninstall)` then `engine.Apply()`.
4. Burn invokes `msiexec /x GraceKeeper-0.2.0.msi /qn` silently. Inner MSI's `StopDismisser` + `RemoveScheduledTask` custom actions fire as today.
5. If "Remove all data" was checked, BA recursively deletes `%ProgramData%\GraceKeeper\` after the MSI completes.
6. BA shows UninstallFinish page. Exits on Finish.

### Dismisser ownership

The dashboard (`GraceKeeper.exe`) becomes the **sole owner** of the dismisser process lifecycle. Three changes flow from this:

- **Removed from MSI:** `LaunchDismisser`, `LaunchDashboard`, `StopDashboard` custom actions. The `GraceKeeperDismisser` HKLM Run entry is removed. The single `GraceKeeper` HKLM Run entry stays.
- **Kept in MSI:** `StopDismisser` custom action (taskkill `AutoHotkey64.exe`) at uninstall — needed because the bundle UI cannot reach into the user session reliably during the MSI execution phase.
- **Dashboard startup logic:** on `App.Startup`, the dashboard calls `DismisserSupervisor.EnsureRunning()`:
  - Reads `%ProgramData%\GraceKeeper\dismisser-pid.json` if it exists.
  - If JSON exists and `Process.GetProcessById(pid)` succeeds and the process's full exe path matches `[INSTALLDIR]\AutoHotkey64.exe` and its command line (via WMI `Win32_Process.CommandLine`) contains `popup-dismisser.ahk` — adopt.
  - Otherwise, enumerate all `AutoHotkey64.exe` processes for orphan recovery. Adopt the first one matching install path + script path.
  - If nothing found, spawn fresh via `Process.Start` with `CREATE_NEW_PROCESS_GROUP` (detached child). Persist PID + exe path + script path + UTC start time to `dismisser-pid.json`.

The supervisor's poll timer (30s) runs while the dashboard is alive. If the persisted PID's process is gone (and no `DISABLED` sentinel exists), spawn fresh and rewrite the JSON.

### `dismisser-pid.json` schema

```json
{
  "pid": 12345,
  "exe_path": "C:\\Program Files\\GraceKeeper\\AutoHotkey64.exe",
  "script_path": "C:\\Program Files\\GraceKeeper\\popup-dismisser.ahk",
  "started_utc": "2026-05-16T20:35:12.482Z"
}
```

Written under the existing `Users : GenericAll` ACL on `%ProgramData%\GraceKeeper\`, same as `config.json`.

## Components

### `GraceKeeper.Bootstrapper`

- Target framework: `.NET Framework 4.8` (matches dashboard).
- Project type: WPF executable produced as a class library DLL consumed by Burn (`BootstrapperApplication` attribute on the entry class).
- Dependency: `WixToolset.Mba.Core` ≥ 5.0 (NuGet).
- Communicates with Burn engine via `IBootstrapperEngine`. Subscribes to `DetectBegin`, `PlanComplete`, `ExecuteMsiMessage`, `Progress`, `ApplyComplete`.
- Pages are simple WPF `UserControl`s swapped into the main `ContentControl`. View-model first; no code-behind beyond `InitializeComponent`.
- All artwork inlined as `Resource` build action. Theme XAML in `Themes/Theme.xaml`.

### `GraceKeeper.Core.Supervisor`

- `DismisserSupervisor` class — exposes `EnsureRunning()`, `Stop()`, and an `OnRestart` event for logging.
- `DismisserPidFile` — read/write/atomic-replace JSON, thread-safe.
- WMI cmdline lookup encapsulated in `ProcessCommandLineReader` (testable abstraction).
- Tests in `GraceKeeper.Core.Tests` extended: PID-adoption logic, orphan-discovery edge cases, JSON corruption recovery.

### `installer/Bundle.wxs`

```xml
<Bundle Name="GraceKeeper" Version="$(Version)" Manufacturer="Nick Cason"
        UpgradeCode="{NEW-GUID}" IconSourceFile="...gracekeeper.ico"
        DisableRemove="no" DisableModify="yes">
  <BootstrapperApplication>
    <Payload SourceFile="..\src\GraceKeeper.Bootstrapper\bin\Release\net48\GraceKeeper.Bootstrapper.dll"/>
    <!-- + dependent DLLs -->
  </BootstrapperApplication>
  <Chain>
    <MsiPackage SourceFile="GraceKeeper-$(Version).msi" Vital="yes" Visible="no" Compressed="yes"/>
  </Chain>
</Bundle>
```

The `BootstrapperApplication` element's `Id` references the managed BA factory class. Exact incantation per WiX 5 + WixToolset.Mba.Core docs.

### `installer/Product.wxs` deltas

- Delete `<CustomAction Id="LaunchDismisser">`, `<CustomAction Id="LaunchDashboard">`, `<CustomAction Id="StopDashboard">`.
- Delete the `<Custom Action="LaunchDismisser" ...>`, `<Custom Action="LaunchDashboard" ...>`, `<Custom Action="StopDashboard" ...>` sequence entries.
- Delete the `GraceKeeperDismisser` registry value inside `RunRegistry` component.
- Bump version (driven by build script).

### `scripts/build-local.ps1` deltas

Pipeline becomes:
1. Restore + build `GraceKeeper.UI`, `GraceKeeper.Core`, `GraceKeeper.Bootstrapper` (Release/net48).
2. Run `GraceKeeper.Core.Tests`.
3. Run Pester suite.
4. Compile dismisser via Ahk2Exe (existing).
5. `wix build` on `installer/Product.wxs` → produces `bin\GraceKeeper-<ver>.msi` (staging).
6. `wix build` on `installer/Bundle.wxs`, supplying the MSI as a payload bindpath → produces `GraceKeeper-<ver>.exe` at repo root.
7. Move the standalone MSI to a `bin/dev/` staging dir (kept locally, not published).

### CI changes

- `.github/workflows/build.yml`: add `dotnet build` step for the bootstrapper project; add bundle build step after MSI build.
- `.github/workflows/release.yml`: replace `*.msi` upload with `*.exe` upload. Keep `*.wixpdb` for both MSI and bundle for diagnostics.

## Error Handling

- **MSI install failure inside Burn.** BA receives `ApplyComplete` with a non-zero status. Progress page transitions to an error page with the Win32 exit code, a "View log" button (opens the Burn log directory), and a "Close" button. Bundle exits with the same code.
- **BA exception.** Burn catches and writes to its log; the user sees a generic "Setup failed" dialog from Burn itself. Acceptable for v0.2.0.
- **Dashboard launch fails on Finish.** Logged; Finish page still closes. The HKLM Run entry guarantees dashboard starts at next login as a fallback.
- **Supervisor cannot spawn dismisser.** Logged to `%ProgramData%\GraceKeeper\logs\supervisor.log` with the OS error code. Dashboard continues running; tray icon shows a warning state (out of scope to design here — minimal indicator: tooltip says "Dismisser not running").
- **Corrupt `dismisser-pid.json`.** Treated as missing. Fall through to orphan discovery, then to fresh spawn.
- **Upgrade from v0.1.0-rc3 → v0.2.0.** MSI's existing `MajorUpgrade` removes the old install before the new one is laid down. The bundle's own `MajorUpgrade` removes any prior bundle if installed. Preserved: `%ProgramData%\GraceKeeper\` contents (counters, config).

## Testing

### Unit (`GraceKeeper.Core.Tests`)

New cases for the supervisor:
- Adopt by matching install path + script path.
- Orphan discovery picks the first matching process when JSON is missing.
- Corrupt JSON → recovery via discovery.
- PID belongs to a different `AutoHotkey64.exe` (different install path) → not adopted, spawn fresh.
- Process exited between read and adoption attempt → spawn fresh.

### Integration / VM (`docs/release-checklist.md`)

Replace the current v0.1.0 install steps with:
1. **Bundle smoke** — double-click `GraceKeeper-0.2.0.exe` on Win10 and Win11 VM snapshots. Confirm SmartScreen warning (expected unsigned), then BA renders dark crimson with all three pages legible at 100 % / 125 % / 150 % DPI.
2. **Launch on finish** — both `GraceKeeper.exe` and `AutoHotkey64.exe` present in Task Manager within 5 s of clicking Finish with "Launch GraceKeeper now" checked. No logout/login required.
3. **Real popup test** — unchanged from v0.1.0 (LogixDesigner → trigger popup → dismissed within 1 s).
4. **Real cleaner test** — unchanged from v0.1.0.
5. **Supervisor kill+respawn** — kill `AutoHotkey64.exe` via Task Manager. Within 30 s a new instance appears; `dismisser-pid.json` reflects the new PID; `supervisor.log` records `respawned pid=...`.
6. **Detached-child survival** — kill `GraceKeeper.exe`. `AutoHotkey64.exe` continues running. Relaunch the dashboard from Start Menu; supervisor.log records `adopted existing dismisser pid=...`.
7. **Bundle ARP uninstall** — Settings → Apps → GraceKeeper → Uninstall. BA opens, confirm, complete. `C:\Program Files\GraceKeeper\` and the scheduled task are gone. `%ProgramData%\GraceKeeper\` persists (default) or is gone (if "Remove all data" was checked).
8. **Upgrade from v0.1.0-rc3** — install rc3, drop a marker file in ProgramData, install v0.2.0 over the top via the bundle, confirm marker survives, dashboard relaunches cleanly.
