# v0.3.0 — Animations & Polish Pass

**Date:** 2026-05-17
**Scope:** Dashboard (GraceKeeper.UI) + Installer/Bootstrapper (GraceKeeper.Bootstrapper) visual polish before first real release. No new features, no protocol changes — purely UI motion and presentation.

## Goal

Replace the current "values just appear" and "progress bar leaps from 0 to 50 to 100" feel with a coherent, paced animation vocabulary that reads as confident and intentional. The brand register is **confident showcase**: motion is unmistakably present but never feels playful or toy-like.

## Design Pillars

1. **Smooth never teleports.** Any value the user perceives as continuous (counters, progress bar) tweens with easing, never jumps.
2. **State changes have an event.** Discrete state changes (status pill flip, phase complete, cleaner finish) get a small but distinct motion — fade, pulse, or check-mark draw-in.
3. **Motion is informational.** The status pulse means "alive and healthy." The phase spinner means "this is the step running right now." Decorative motion (e.g. shimmer on the progress bar) is reserved for things actively in-flight.
4. **Consistency over cleverness.** All durations and easings come from a shared set of theme tokens. No one-off animation values.

## Theme Tokens (shared)

Added to both `GraceKeeper.UI/Themes/Dark.xaml` (and `Light.xaml`) and `GraceKeeper.Bootstrapper/Themes/Theme.xaml`:

| Key | Value | Use |
|---|---|---|
| `Duration.Quick` | 120 ms | Button press feedback, micro-state flips |
| `Duration.Standard` | 240 ms | Page transitions, fade-ins, phase row state changes |
| `Duration.Slow` | 480 ms | Welcome hero cascade, log entry slide-in tail |
| `Duration.Counter` | 1200 ms | Counter roll, installer-bar catch-up |
| `Duration.Pulse` | 1600 ms | Status dot opacity pulse (autoreverse) |
| `Easing.Standard` | `CubicEase { EasingMode = EaseOut }` | Default for all tweens |
| `Easing.Decelerate` | `QuinticEase { EasingMode = EaseOut }` | Counter roll (slow finish makes the landing feel weighty) |

Everything below references these tokens — implementation must not hard-code durations or easings except where explicitly noted.

## Dashboard (GraceKeeper.UI)

### `AnimatedCounter` control

New file: `src/GraceKeeper.UI/Controls/AnimatedCounter.cs` + `AnimatedCounter.xaml`.

- Templated `Control` (not UserControl — we want consumers to retemplate font/size).
- Dependency properties:
  - `Value` (long) — input target.
  - `DisplayValue` (double) — internal, used by the template's TextBlock via a converter that formats `(long)DisplayValue` as `N0` (thousands separator, locale-aware).
  - `RollDuration` (Duration, default = `Duration.Counter`).
  - `Easing` (IEasingFunction, default = `Easing.Decelerate`).
- Behavior: when `Value` changes, animate `DisplayValue` from its current value to the new `Value` via `DoubleAnimation` using `RollDuration` and `Easing`. The animation always starts at the current rendered value, not the previous `Value` — so a Value change mid-animation retargets smoothly.
- **On first load**, `DisplayValue` starts at 0 and animates up to the loaded `Value`. This matches the chosen "roll from 0 on every open" behavior.
- Renders a single `TextBlock`; consumers control font size via TextBlock-style inheritance through the template.

### Dashboard view changes

`src/GraceKeeper.UI/Views/DashboardView.xaml`:

1. **Hero metric cards** — Replace the three large `<TextBlock Text="{Binding PopupsDismissedLifetime}" FontSize="28" ...>` with `<ctl:AnimatedCounter Value="{Binding PopupsDismissedLifetime}" FontSize="28" FontWeight="SemiBold"/>`. Same for the `.rnl files deleted` card. The third card (`TimeSinceLastCleanText`) is a string — stays as TextBlock. The "+N today" sub-text on each card is also wrapped in an `AnimatedCounter` (smaller, no flourish — 600ms duration override).

2. **Recent activity list** — Add an `ItemContainerStyle` with:
   - Initial `Opacity = 0`, `RenderTransform = TranslateTransform(0, -8)`.
   - `EventTrigger RoutedEvent="Loaded"` runs a parallel storyboard: opacity 0→1 over `Duration.Standard` and `TranslateTransform.Y` -8→0 over `Duration.Slow`, both with `Easing.Standard`. Result: each new log entry slides down and fades in.
   - The existing `Insert(0, ev)` pattern in `DashboardViewModel.RefreshFromLogs` already triggers `Loaded` for newly-inserted items, so this just works.

3. **Status pill** — The dot `<Ellipse>` gets a `Style.Triggers` block:
   - When the status text equals `"Healthy"`, a `DoubleAnimation` on `Opacity` from 1.0 → 0.7 → 1.0 over `Duration.Pulse`, `AutoReverse = True`, `RepeatBehavior = Forever`.
   - When status is `"Paused"`, opacity is static 1.0 (no pulse — paused should feel still).
   - Implementation note: WPF Style triggers can't bind on string equality directly via `DataTrigger` to `StatusText` *and* control a storyboard cleanly across all paths — use a `Storyboard.EnterActions`/`ExitActions` pattern on a `DataTrigger Value="Healthy"`.

4. **Run cleaner button** — Replace the existing `<Button Content="Run cleaner now" .../>` with a new templated `RunCleanerButton` control (see below).

### `RunCleanerButton` control

New file: `src/GraceKeeper.UI/Controls/RunCleanerButton.cs` + `RunCleanerButton.xaml`.

- Templated `Control` deriving from `ButtonBase` (so `Click` event works idiomatically).
- Dependency property: `State` (enum `CleanerButtonState { Idle, Running, Done }`).
- Template visual layout: a `Grid` with a `Border` (background = Brand.Red), a `TextBlock` for the label, and a circular spinner (a `Path` of an arc with a `RotateTransform` animated 0→360 over 900ms `RepeatBehavior=Forever`).
- VisualStateManager groups:
  - **CommonStates**: Normal, MouseOver, Pressed (existing button styling).
  - **CleanerStates**: Idle, Running, Done.
    - `Idle → Running`: label text changes to "Cleaning…", spinner fades in (opacity 0→1) and starts rotating, button width slightly contracts (~10%) via `LayoutTransform` scale, click is disabled.
    - `Running → Done`: spinner fades out, label flashes to "Done ✓" with a brief `Foreground` color pop (Semantic.Good), background tints Semantic.Good for ~600ms.
    - `Done → Idle`: 1200ms auto-revert (timer in code-behind), label and background ease back to "Run cleaner now" + Brand.Red.
- All transitions use `Duration.Standard` with `Easing.Standard` except the spinner rotation (linear) and the Done auto-revert delay (`Duration.Counter`).

### `DashboardViewModel` changes

`src/GraceKeeper.UI/ViewModels/DashboardViewModel.cs`:

- Add `public CleanerButtonState CleanerState { get; private set; }` with `OnPropertyChanged`.
- Rework `RunCleanerNowAsync`:
  ```
  CleanerState = Running;
  var startOffset = current cleaner-log offset;
  await _scheduler.RunNowAsync();
  // poll for a new clean-kind log entry, up to 60s timeout
  while (timeout not elapsed) {
      await Task.Delay(500);
      if (cleaner log has new clean entry past startOffset) break;
  }
  CleanerState = Done;
  // Done auto-reverts visually after Duration.Counter; reset model state on a 1200ms timer too
  ```
- `DashboardView.xaml.cs` `RunCleaner_Click` continues to call `Vm.RunCleanerNowAsync()`; the button's `State` is bound to `Vm.CleanerState`.

## Installer / Bootstrapper (GraceKeeper.Bootstrapper)

### `PhaseList` control

New file: `src/GraceKeeper.Bootstrapper/Controls/PhaseList.xaml` (+ code-behind).

- UserControl exposing `Phases` (ObservableCollection of `PhaseRow`).
- `PhaseRow` model: `string Title`, `PhaseState State` (Pending/Active/Done), with INotifyPropertyChanged.
- `ItemsControl` with a row template:
  - Left: a 16×16 indicator. VSM controls its appearance:
    - **Pending**: gray dot (radius 3, Border.Default color).
    - **Active**: animated spinner (rotating arc, same as `RunCleanerButton`'s spinner).
    - **Done**: check-mark `Path` that draws in via a `StrokeDashArray` animation over `Duration.Standard`.
  - Right: row label. Text color via VSM:
    - Pending: MutedText.
    - Active: Text (white).
    - Done: MutedText again (calmer, done is done).
- Transitions between states animate over `Duration.Standard` with `Easing.Standard` for color/opacity; the check-mark draw is its own short animation.

### `ProgressPage` changes

`src/GraceKeeper.Bootstrapper/Views/ProgressPage.xaml`:

1. **Smoothed progress bar** — The `ProgressBar`'s `Value` binds to `SmoothedPercent` (not `Percent`). `SmoothedPercent` tweens toward `Percent` whenever `Percent` changes (see ViewModel).
2. **Shimmer overlay** — On top of the `ProgressBar`, layer a transparent `Rectangle` filled with a `LinearGradientBrush`. The gradient has three stops: transparent → semi-opaque white (alpha ~0.18) → transparent. A storyboard animates the `Offset` of the middle stop from -0.3 to 1.3 over 1800ms, `RepeatBehavior=Forever`. The shimmer is only visible while the installer is actively running (bound visibility on a `IsRunning` flag — true between Plan start and ApplyComplete).
3. **Phase list** — Below the bar, mount `<ctl:PhaseList Phases="{Binding Phases}" Margin="0,18,0,0"/>`. The phase list replaces the current single "Installing…" line; the existing per-MSI `StatusText` line stays underneath the phase list as small muted text (the verbatim Burn message, for power users).

### `MainViewModel` (bootstrapper) changes

`src/GraceKeeper.Bootstrapper/ViewModels/MainViewModel.cs`:

- New properties:
  - `double SmoothedPercent` — backing field tweened by a `DispatcherTimer` (60Hz) using a critically-damped follow toward `Percent`. Simpler alternative: hold a `DoubleAnimation` reference and retarget on each `Percent` setter (`new DoubleAnimation(_smoothedPercent, Percent, Duration.Counter)`). Use the latter — fewer moving parts.
  - `ObservableCollection<PhaseRow> Phases` — initialized at construction with three rows: Preparing, Installing, Finishing. (Uninstall uses the same three labels — semantics fit either direction; copy is intentionally generic.)
  - `bool IsRunning` — true between Plan start and ApplyComplete; drives shimmer visibility.
- Phase transition rules:
  - `DoInstall` / `DoUninstall`: `Phases[0].State = Active`; `IsRunning = true`.
  - `OnPlanComplete` (success): `Phases[0].State = Done`; `Phases[1].State = Active`.
  - `OnProgress` with `OverallPercentage >= 100`: `Phases[1].State = Done`; `Phases[2].State = Active`.
  - `OnApplyComplete` (success): `Phases[2].State = Done`; `IsRunning = false`; brief pause (300ms) before navigating to FinishPage to let the user see the final check-mark land.

### `WelcomePage` polish

`src/GraceKeeper.Bootstrapper/Views/WelcomePage.xaml`:

- The hero line `Keep the grace period alive.` is decomposed into individual `<Run>` elements per word inside a single `TextBlock`. Wrap each word in a small `TextBlock` (replacing the `Run`s) with `Opacity = 0` and a translation offset, then animate them in on page load with a 60ms stagger:
  - Word 1 ("Keep"): start at t=0
  - Word 2 ("the"): start at t=60ms
  - …etc, each word fading from 0→1 over `Duration.Slow` with a `TranslateTransform.Y` from 8→0.
- All other welcome page elements remain static.

### Page transitions

`src/GraceKeeper.Bootstrapper/MainWindow.xaml`:

- Wrap `<ContentControl Content="{Binding CurrentPage}"/>` in a custom `ContentControl` style that runs a fade-out (`Duration.Quick`) on the outgoing content followed by a fade-in + 6px slide-up on the incoming content (`Duration.Standard`). Implementation: override `ContentControl`'s template with a single `ContentPresenter` whose `Opacity` and `RenderTransform.Y` animate on `Content`-changed via `EventTrigger` on `Loaded`. The existing `CurrentPage` swap pattern in `MainViewModel` already produces a fresh element each time, so each new page raises `Loaded` and runs the entry storyboard.

## Testing Plan

Animations themselves can't be unit-tested without a UI test harness (we don't have one). Tests cover the underlying state:

1. `AnimatedCounterTests` (new) — set Value, immediately query DisplayValue end-state (using `BeginAnimation` with the SnapshotAndReplace fill behavior we let it complete via a synchronous dispatcher flush). Verify the final rendered string matches the formatted target.
2. `MainViewModelTests` (new) — phase advancement on simulated Detect / PlanComplete / Progress / ApplyComplete events. SmoothedPercent eventually equals Percent (advance a stub timer).
3. `RunCleanerButtonTests` — state transition contract: Done auto-reverts to Idle after the configured delay.

Manual QA checklist (in `docs/release-checklist.md`):

- Open dashboard: counters roll up from 0; status dot pulses.
- Trigger a popup externally; new log entry slides in.
- Click Run cleaner: button morphs to spinner; on completion, label flashes "Done ✓" then reverts.
- Run installer: progress bar moves smoothly through 0-50-100; phase rows advance Preparing → Installing → Finishing with the spinner moving from row to row; final check-mark lands before the page transitions to Finish.

## Risks & Mitigations

- **Animation performance on low-end hardware.** WPF storyboards on the UI thread can stutter under load. Mitigation: keep concurrent animations small (most are single-property), use `RenderOptions.BitmapScalingMode=LowQuality` only if needed, profile during QA.
- **Cleaner-poll never sees a completion entry.** If the cleaner crashes or the log is rotated, `RunCleanerNowAsync` would hang at Running. Mitigation: 60s hard timeout falls back to Idle (not Done — Done implies success).
- **Shimmer animation leaks.** A `RepeatBehavior=Forever` storyboard on a non-visible element wastes cycles. Mitigation: bind shimmer visibility to `IsRunning`; collapsed visibility halts the storyboard naturally in WPF.
- **Burn page transition timing.** The 300ms pause before Finish navigation must not feel laggy; if QA reports it does, drop to 150ms or remove entirely.

## Out of Scope

- Sound effects.
- Particle effects, confetti, celebratory bursts.
- Per-digit "slot machine" counter animation (we picked whole-number tween).
- Light theme tuning beyond the new shared tokens — existing Light.xaml stays, just inherits the new keys.
- Animated icons (SVG or otherwise) — we draw spinners and check-marks with WPF `Path` geometries inline.
- Reduced-motion preference detection. Could be added later as a setting; not in this pass.

## Acceptance Criteria

- Dashboard open: counters animate from 0 to current value over 1.2s; status dot pulses if healthy; no metric appears via teleport.
- Run cleaner from dashboard: button visibly transforms during the run; reverts to default after completion or 60s timeout.
- Installer: progress bar never jumps more than 1% per 16ms; phase list advances in lockstep with Burn lifecycle; welcome hero animates in on page mount; page changes fade+slide.
- All durations/easings reference shared theme tokens (grep should find no hard-coded `Duration="00:00:`-style values in views except inside the theme dictionaries).
- All existing tests still pass; new tests for AnimatedCounter / MainViewModel phases / button state machine pass.
