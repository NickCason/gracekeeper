# v0.4.0 — Auto .rnl Cleanup That Actually Runs

**Date:** 2026-05-26
**Scope:** Replace the SYSTEM-context PowerShell scheduled task with a tray-app-driven cleanup pipeline, fix the schtasks XML defaults that silently swallowed missed runs, and add a boot-time always-clean window plus defense-in-depth for Echo-locked .rnl files.

## Why v0.4 Exists

A user with v0.3.x running "for days" reported that `.rnl` files in `C:\ProgramData\Rockwell Automation\FactoryTalk Activation\` were going stale until they manually pressed *Run cleaner now*. Investigation found two independent root causes:

### Bug 1 — The schedule never recovers missed runs

The MSI creates the cleanup task via:

```
schtasks.exe /Create /F /TN "GraceKeeper - Cleanup RNL"
             /TR "powershell.exe ... rnl-cleaner.ps1"
             /SC HOURLY /MO 12 /RU SYSTEM /ST 03:00
```

`schtasks.exe`'s CLI defaults bake in three task-XML settings that are wrong for a background maintenance job on a workstation:

| Setting | schtasks default | Effect |
|---|---|---|
| `<DisallowStartIfOnBatteries>` | `true` | Silently skipped on any laptop on battery |
| `<StopIfGoingOnBatteries>` | `true` | Killed mid-run on power transition |
| `<StartWhenAvailable>` | *(absent → false)* | **Missed runs are dropped forever** |
| `<WakeToRun>` | *(absent → false)* | Will not wake at 03:00 (correct, but combined with no `StartWhenAvailable`, missed runs never catch up) |

A workstation that sleeps overnight misses the 03:00 fire and never gets it back. If lunch lid-close happens at 15:00, both daily windows are gone. Live evidence on the dev machine: the task was registered at 09:14:15 today, `cleaner.log` contains exactly **one** entry — from a manual button press at 09:52:57. No automatic run has fired since install.

### Bug 2 — Locked files are dropped silently with no remediation

`rnl-cleaner.ps1` handles per-file failure with:

```powershell
try { Remove-Item $full -Force -ErrorAction Stop; $deleted++ }
catch { $lockedNames += $name }
```

A file that's locked once stays locked on every subsequent run unless whatever is holding it happens to release between cycles. There is no retry, no service-stop fallback, and the dashboard surfaces nothing beyond a log line. The user's instinct — that Logix Echo's service holds .rnl locks — is partially correct (see *Empirical Findings That Reshape the Design* below).

## Empirical Findings That Reshape the Design

Tested live on a fully-provisioned Rockwell SI workstation (Studio 5000 + FactoryTalk + Logix Echo) with **one Echo controller actively emulating a CompactLogix 5380**:

1. **Process-tree detection is clean and free.** `FactoryTalk Logix Echo Service` = `EmulateService.exe`; it spawns one child per active controller, named `Emulate{Family}.exe` (e.g. `EmulateCompactLogix5380.exe`). Counting child processes via `Win32_Process.ParentProcessId` gives an authoritative active-controller count and the controller family name — no Echo REST API needed.

2. **.rnl files are usually unlocked, even during active Echo emulation.** Probing each existing `.rnl` with `File.Open(... FileShare.None)` while Echo had a running controller: all three opened cleanly. The Activation Service (`lmgrd.exe` — Flexera FlexLM) holds handles transactionally (open → read/write → close), not persistently.

3. **Triggering cleanup mid-emulation succeeded without any service bounce:**

   | | Pre-clean | Post-clean (T+~40s) |
   |---|---|---|
   | File count | 3 | 3 (regenerated) |
   | Filenames | `4330c41b`, `ae29d238`, `fdd0ce35` | same three (filenames are deterministic per activation product) |
   | SHA256 prefix | `0EF4…`, `A4BA…`, `7C99…` | `935D…`, `CC69…`, `89C4…` — all rotated |
   | Echo controller PID 14580 | running | **still running, never restarted** |
   | Service bounce needed | — | **no** |

   Conclusion: the .rnl filename is stable per activation product; the *grace-period token inside the file* rotates on regeneration. Deleting a live .rnl does **not** fault Echo's controller — the activation service reissues a fresh lease on the next client poll (~9-12s per product, three rounds for three products).

4. **Re-interpretation of the "Logix Echo .rnl quirk":** the user's empirical experience is real but rare — Echo's service does occasionally hold an exclusive lock during activation handshakes (tens of milliseconds). A simple **retry-with-backoff** catches >99% of these. A full service bounce is a true last resort, not the first escalation tier.

## Design Goals

1. **Cleanup that actually runs.** Defeat the schtasks-defaults failure modes. Multiple independent triggers so any one failing still leaves the system clean.
2. **Single source of truth for cleanup logic.** Replace `rnl-cleaner.ps1` with C# in `GraceKeeper.Core`. Three triggers, one code path.
3. **Never break user work.** Default mode never bounces services if Echo controllers are running. Service bounce is reserved for boot-time (no user yet) and explicit user opt-in.
4. **Self-evident from the dashboard.** Users can see when cleanup ran, what was refreshed, and whether anything was deferred and why.

## Non-Goals

- No support for non-Rockwell activation systems.
- No remediation if the FT Activation Service itself is faulted — that's outside our scope; we just clean files.
- No Echo REST API integration in v0.4. Process-tree detection is sufficient.
- No code signing of the new console exe in v0.4. (Tracked separately — same SmartScreen story as the main installer.)

## Architecture

### Three triggers, one code path

```
┌─ Boot (every Windows boot, before user logon) ────────────────┐
│  GraceKeeper - Boot Cleanup    (NEW)                          │
│    Task Scheduler <BootTrigger> / <Delay>PT0S</Delay>         │
│    Runs as: NT AUTHORITY\SYSTEM                               │
│    Invokes: GraceKeeper.Cleaner.exe --mode boot               │
└────────────────────────────────────────────────────────────────┘

┌─ Runtime (in tray-app process, while user is working) ────────┐
│  DispatcherTimer inside GraceKeeper.exe                        │
│    Fires: T+30s after app launch, then every IntervalHours    │
│    Runs in: user's session, in-process via Core                │
│    Invokes: RnlCleaner.RunAsync(CleanupMode.Runtime)          │
└────────────────────────────────────────────────────────────────┘

┌─ Safety net (every IntervalHours regardless of tray state) ──┐
│  GraceKeeper - Cleanup RNL    (KEPT, XML rewritten)          │
│    Task Scheduler <TimeTrigger> / Repetition.Interval=PT12H  │
│    Runs as: NT AUTHORITY\SYSTEM                              │
│    Invokes: GraceKeeper.Cleaner.exe --mode safety-net        │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
              All three call the same logic in
              GraceKeeper.Core.RnlCleaner
              behavior gated by CleanupMode
```

## Cleanup modes

`CleanupMode` is a Core enum:

| Mode | Trigger | Echo-busy behavior | Service-bounce behavior |
|---|---|---|---|
| `Boot` | Boot scheduled task | Ignored (no user session, no controllers will be running this early) | Always bounce if any lock survives 3 retry attempts |
| `Runtime` | Tray-app DispatcherTimer + manual button (default) | If any Echo controller is running, do not bounce services. Delete what's unlocked, defer the rest. | Bounce only when Echo is idle. |
| `SafetyNet` | SYSTEM-context Cleanup RNL task | Same as Runtime, plus: skip entirely if `cleaner.log` shows a `deleted=` line within the last 11 hours (avoids double-cleaning when the tray was already alive). | Same as Runtime. |
| `ManualForce` | Dashboard button + "Run anyway" confirm | Bypass Echo-busy guard. (User explicitly accepted controller fault.) | Bounce regardless of Echo state. |

### Cleanup execution flow (all modes)

```
1. Read DISABLED sentinel. Present → log "skipped (disabled)", exit.
2. (SafetyNet only) Check cleaner.log tail. If a successful run is < 11h old, exit.
3. Enumerate target dir for *.rnl. Empty → log "no files", exit.
4. For each file:
     attempt 1: Remove(). Success → tally.
     attempt 2 (200ms later): Remove(). Success → tally.
     attempt 3 (400ms later): Remove(). Success → tally.
     all three failed → add to LockedFiles list.
5. If LockedFiles is empty:
     Log: "refreshed=N | deferred=0 | duration=Xms"
     Exit success.
6. Decide bounce eligibility:
     mode == Boot           → eligible
     mode == ManualForce    → eligible
     mode in {Runtime, SafetyNet}:
       EchoControllerProbe.GetActiveCount() == 0 → eligible
       else → not eligible
7. Not eligible → log:
     "refreshed=N | deferred=M (echo-busy: <familyA, familyB>) | duration=Xms"
     Exit success (the cleanup partially succeeded).
8. Eligible → ServiceBouncer.BounceAndRetryAsync(LockedFiles):
     a. Stop FactoryTalk Logix Echo Message Broker
     b. Stop FactoryTalk Logix Echo Service
     c. Stop FactoryTalk Activation Service
     d. Stop FactoryTalk Activation Helper (FTActivationBoost)
     e. Wait until all four report Stopped or 10s timeout.
     f. Retry Remove() on LockedFiles. Tally as freed-by-bounce.
     g. Start FactoryTalk Activation Helper
     h. Start FactoryTalk Activation Service
     i. Start FactoryTalk Logix Echo Service
     j. Start FactoryTalk Logix Echo Message Broker
     k. Wait until all four report Running or 10s timeout.
9. Log: "refreshed=N | freed-by-bounce=M | still-locked=K | duration=Xms"
```

The Helper-before-Service start ordering on steps 8.g–8.h follows Rockwell's documented restart sequence (see *Restart the activation services*, `activate.rockwellautomation.com/ActivationHelp/216953.htm`).

### Component contracts

```csharp
namespace GraceKeeper.Core;

public enum CleanupMode { Boot, Runtime, SafetyNet, ManualForce }

public sealed record CleanupResult(
    int RefreshedCount,
    int FreedByBounceCount,
    int DeferredCount,
    IReadOnlyList<string> DeferredFiles,
    string? DeferredReason,      // "echo-busy: CompactLogix5380" etc.
    int StillLockedCount,
    TimeSpan Duration);

public interface IRnlCleaner {
    Task<CleanupResult> RunAsync(CleanupMode mode, CancellationToken ct);
}

public interface IEchoControllerProbe {
    EchoActivity GetActivity();   // (Count, IReadOnlyList<string> FamilyNames)
}

public interface IServiceBouncer {
    Task<BounceResult> BounceAndRetryAsync(
        IReadOnlyList<string> filesToRetry, CancellationToken ct);
}
```

All three interfaces have concrete implementations in Core and fakes in `GraceKeeper.Core.Tests` so the orchestration logic in `RnlCleaner` is fully unit-tested without touching the live machine.

## Scheduled task XML — the corrected version

Both `GraceKeeper - Cleanup RNL` and `GraceKeeper - Boot Cleanup` use the same XML skeleton, differing only in `<Triggers>`. Below is the safety-net task; the boot task swaps the `<TimeTrigger>` for a `<BootTrigger>`.

```xml
<?xml version="1.0" encoding="UTF-16"?>
<Task version="1.2" xmlns="http://schemas.microsoft.com/windows/2004/02/mit/task">
  <Principals>
    <Principal id="Author">
      <UserId>S-1-5-18</UserId>           <!-- LocalSystem -->
      <RunLevel>HighestAvailable</RunLevel>
    </Principal>
  </Principals>
  <Settings>
    <DisallowStartIfOnBatteries>false</DisallowStartIfOnBatteries>
    <StopIfGoingOnBatteries>false</StopIfGoingOnBatteries>
    <StartWhenAvailable>true</StartWhenAvailable>
    <WakeToRun>false</WakeToRun>
    <AllowHardTerminate>false</AllowHardTerminate>
    <ExecutionTimeLimit>PT15M</ExecutionTimeLimit>
    <MultipleInstancesPolicy>IgnoreNew</MultipleInstancesPolicy>
    <IdleSettings>
      <StopOnIdleEnd>false</StopOnIdleEnd>
      <RestartOnIdle>false</RestartOnIdle>
    </IdleSettings>
    <Priority>7</Priority>
  </Settings>
  <Triggers>
    <TimeTrigger>
      <StartBoundary>2026-01-01T03:00:00</StartBoundary>
      <Repetition>
        <Interval>PT12H</Interval>
        <!-- No Duration → repeats indefinitely -->
        <StopAtDurationEnd>false</StopAtDurationEnd>
      </Repetition>
    </TimeTrigger>
  </Triggers>
  <Actions Context="Author">
    <Exec>
      <Command>"C:\Program Files\GraceKeeper\GraceKeeper.Cleaner.exe"</Command>
      <Arguments>--mode safety-net</Arguments>
    </Exec>
  </Actions>
</Task>
```

Critical lines vs. the broken v0.3 task:

- `<DisallowStartIfOnBatteries>false</...>` and `<StopIfGoingOnBatteries>false</...>` — laptop users get cleanups.
- `<StartWhenAvailable>true</...>` — **missed runs catch up when the machine wakes**. This is the single most important XML change.
- `<ExecutionTimeLimit>PT15M</...>` — bounded runaway protection.
- `<WakeToRun>false</...>` — explicitly off. We don't wake the machine at 03:00 just to clean files; that would be obnoxious. `StartWhenAvailable` is sufficient.

### Boot trigger variant

```xml
<Triggers>
  <BootTrigger>
    <Delay>PT0S</Delay>
  </BootTrigger>
</Triggers>
```

`<Delay>PT0S</Delay>` (zero delay) means "fire as soon as the task service is up." The boot trigger commonly races the FT services for who reaches the file system first; that's fine — `--mode boot` is permitted to bounce services unconditionally, so whether they're up at trigger time or not, we converge on "files cleaned, services running."

## Installer changes (`installer/Product.wxs`)

1. Add `GraceKeeper.Cleaner.exe` as a Component under `INSTALLDIR`.
2. Delete the `CleanerPs1` Component (and the `rnl-cleaner.ps1` file) entirely.
3. Replace the inline `schtasks.exe /Create /F /SC HOURLY ...` custom action with two PowerShell custom actions that register tasks from inline XML (mirroring how `register-logon-task.ps1` already works for the dashboard logon task). This is the only reliable way to set non-default `<StartWhenAvailable>` etc. — schtasks.exe doesn't expose them.

   New script: `src/register-cleaner-tasks.ps1`. Registers both `GraceKeeper - Cleanup RNL` and `GraceKeeper - Boot Cleanup` from heredoc XML. Removal action stays as `schtasks /Delete /F /TN "<name>"`.
4. The existing `CreateScheduledTask` custom action is retired; the new `register-cleaner-tasks.ps1` is invoked via a single custom action and registers both tasks atomically.

## Dashboard changes (`GraceKeeper.UI`)

Minimal surface — the dashboard already shows lifetime/today counters and a recent activity log. v0.4 adjustments:

- The "RNL files deleted" counter is renamed to **"RNL files refreshed"** in XAML strings. The number is unchanged (still parsed from `refreshed=N` log lines, which were `deleted=N` pre-v0.4). The `cleaner.log` schema bumps to include both for one release so users upgrading mid-stream don't see counter resets — `DashboardViewModel._deletedCountRegex` matches both `deleted=(\d+)` and `refreshed=(\d+)`.
- Recent activity list distinguishes three event variants:
  - `refreshed=N` → "Refreshed N files" (normal case)
  - `deferred=M (echo-busy: X)` → "Deferred M files (Echo busy: X)" — distinct icon
  - `freed-by-bounce=M` → "Bounced services to refresh M files" — distinct icon
- Manual *Run cleaner now* button:
  - If `EchoControllerProbe.GetActivity().Count == 0` → invoke mode `Runtime` immediately, no dialog.
  - If `Count > 0` → show modal: *"Echo has N active controller(s) ({family list}). Cleanup may fault them if a service bounce becomes necessary. [Cancel] [Run anyway]"*. Cancel returns to idle, no work done. Run anyway invokes mode `ManualForce`.
- No changes to the "Next run" countdown logic structurally — but it now displays the in-process DispatcherTimer's next-fire time, not the schtask's. The settings view's "Cleanup interval" change handler stops calling `ScheduledTaskClient.ChangeIntervalAsync()` and instead updates the in-process timer + persists `IntervalHours` to config; the schtask is re-registered to the same interval at MSI install time only.

## DispatcherTimer in the tray app

```csharp
// GraceKeeper.UI/App.xaml.cs — added after StartDismisserSupervisor()
private void StartCleanupSchedulerOnce()
{
    if (_cleanupScheduler != null) return;
    var cleaner = new RnlCleaner(...);
    _cleanupScheduler = new CleanupScheduler(
        cleaner,
        intervalProvider: () => _configStore.Load().Cleaner.IntervalHours,
        launchDelay: TimeSpan.FromSeconds(30));
    _cleanupScheduler.Start();
}
```

`CleanupScheduler` is a thin wrapper around `DispatcherTimer`:
- First tick: 30s after `Start()` (the "launch clean" — gives FT services time to settle after logon).
- Subsequent ticks: every `IntervalHours` hours.
- Tick handler: invoke `RnlCleaner.RunAsync(CleanupMode.Runtime)` on a background task; do not block the UI thread.
- Reschedules cleanly when the user changes the interval in settings.

## Upgrade migration

Users upgrading v0.3.x → v0.4.0 via the bundled MSI:

1. `StopDashboard` + `StopDismisser` (existing) — kill running processes.
2. `RemoveScheduledTask` (existing, kept) — drops the old `GraceKeeper - Cleanup RNL` task.
3. Files installed: new `GraceKeeper.Cleaner.exe`, new `register-cleaner-tasks.ps1`, updated `GraceKeeper.exe`. The old `rnl-cleaner.ps1` is removed by the absence of its Component (MSI tracks it).
4. New custom action `CreateCleanerTasks` runs `register-cleaner-tasks.ps1`, registering both the rewritten `GraceKeeper - Cleanup RNL` and the new `GraceKeeper - Boot Cleanup`.
5. `CreateDashboardLogonTask` (existing) — unchanged.

`cleaner.log` is untouched across upgrades — counters and history survive. The first post-upgrade `cleaner.log` line emits `refreshed=N` in the new format; the dashboard regex matches both old and new.

## Risks

- **`<BootTrigger>` race with FT services.** The boot task may fire while FT Activation/Echo services are still starting. Mode=Boot is unconditionally allowed to bounce, so this is benign — we either delete cleanly (services not yet attached) or bounce-and-delete (services up but no user work yet). Either path converges.
- **Cleaner.exe failure under SYSTEM context.** Console exits with non-zero on unhandled exception. Task Scheduler logs the failure; the user sees no symptom unless they open Event Viewer. Mitigated by: cleaner.exe wraps `RnlCleaner.RunAsync` in a top-level try/catch that always writes a `failed: <exception type>` line to `cleaner.log` before propagating. The dashboard already tails this file and would surface the failure.
- **Echo introduces a new service in a future version.** Hardcoded service names (`FactoryTalk Logix Echo Service`, `FactoryTalk Logix Echo Message Broker`) become stale. Mitigation: `ServiceBouncer` resolves services by name at runtime; missing services are skipped with a logged warning, not a fatal error.
- **NATS message broker dependency on Echo Service.** Stopping the broker before the service may leave dangling subscribers. Verify during implementation: if `EmulateService.exe` errors when its broker disappears mid-flight, swap the order to Service-then-Broker. Risk is small in mode=Boot (services either both up or both down at trigger time) and moot in mode=Runtime (we only bounce if no controllers are running, so the service has nothing to talk to anyway).

## Out-of-scope items deliberately left for later versions

- Configurable per-product activation policies (e.g. "always defer Echo .rnl, always refresh Studio 5000").
- Echo REST API integration (would replace process-tree probe with a more authoritative signal).
- A "next cleanup at" precise timestamp on the dashboard (currently relative).
- Cleanup metrics export (Prometheus / Telegraf for fleet observability).
- Cross-user notification when a deferred-due-to-Echo file persists across N cycles (a hint that the user has a long-running controller they may have forgotten about).

## Acceptance criteria

A v0.4.0 build is shippable when:

1. Installing v0.4.0 on a workstation that sleeps overnight results in a successful `refreshed=N` log entry within 6 hours of waking, without any manual intervention.
2. Triggering a cleanup with an Echo CompactLogix 5380 actively emulating leaves the controller's process PID unchanged and surfaces a "deferred (echo-busy)" event on the dashboard if files were locked at retry time, OR a `refreshed=N` event if files were not locked.
3. The `GraceKeeper - Boot Cleanup` task fires on the first post-install Windows boot and logs a `refreshed=` (or `freed-by-bounce=`) entry before user logon completes.
4. xUnit coverage of `RnlCleaner`, `EchoControllerProbe`, `ServiceBouncer` ≥ 90% line, with all four `CleanupMode` enumeration paths exercised.
5. Uninstall removes both scheduled tasks and the cleaner.exe binary; reinstall recreates them cleanly with the corrected XML.
